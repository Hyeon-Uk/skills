# Tizen Smoke Test Authoring Skill

Use this skill to add or scaffold **smoke tests** for any package under `~/.openclaw/workspace/gerrit/*`. A smoke test is the narrow, fast, post-install sanity check that runs on a device or emulator immediately after the RPM is installed — its job is to answer "did this package install in a usable state?" in under a few seconds, not to cover the API surface.

This skill has two siblings:

- `tizen-gtest-unit-test` — exhaustive in-process gtest with mocks. Comprehensive coverage.
- `tizen-integration-test` — full end-to-end gtest with real deps and `GMainLoop`. Cross-process flows.

Pick **this skill** when:

- You want a **post-install runner** (`tizen-unittests/<pkg>/run-unittest.sh`) — the standard Tizen CI/QA pattern that 30+ packages in this workspace already follow. This is the default for library packages.
- You're working on an **installer package** (rpk-installer, wgt-backend, tpk-backend, unified-backend, app-installers) where "smoke" means "install a real .wgt/.tpk/.rpk and verify it landed". This is a separate `test/smoke_tests/` C++ gtest binary, modelled on what's already in those packages.
- You need a **single critical-path test** that validates one subsystem without crossing process boundaries (e.g. `capmgr/src/unit_tests/smoke_test.cc` style — DB schema + capability lookup in the same process).

If the test would test many APIs exhaustively → unit test. If it would spin up two processes and exercise cross-component callbacks → integration test. Smoke is intentionally *narrow*.

---

## 0. Pick the pattern (decide first)

Three concrete patterns exist in this workspace. Read this section first, pick one, then jump to the matching phase below.

### Pattern A — `run-unittest.sh` post-install runner (default, 90% of packages)

A POSIX shell script generated by the `.spec` file at install time and dropped at `/usr/bin/tizen-unittests/<pkg>/run-unittest.sh`. It does `setup → test_main → teardown`, where `test_main` invokes the existing `*_unittests` binary (or a `--gtest_filter='*Smoke*'` subset of it). Tizen CI auto-discovers any executable named `run-unittest.sh` under `/usr/bin/tizen-unittests/`, so dropping the file is the entire integration with the test harness — there is no separate registration step.

Reference implementations: `rpc-port/packaging/rpc-port.spec` lines 176–217, `notification/packaging/notification.spec`, `bundle/packaging/bundle.spec`, `aul-1/packaging/aul.spec`.

Use Pattern A when:
- The package already has a `*_unittests` binary you want to gate behind a sanity wrapper.
- You want the standard Tizen CI hook with zero extra build wiring.
- The smoke check is "does the test binary run to completion without crashing?"

### Pattern B — `test/smoke_tests/` C++ gtest binary (installer packages)

A separate C++ gtest binary, built alongside the package, that exercises real install/uninstall flows on a sample artefact. Lives under `<pkg>/test/smoke_tests/` and ships in its own `*-installer-ut` (or similar) subdirectory under `/usr/bin/`.

Reference implementations: `wgt-backend/test/smoke_tests/`, `tpk-backend/test/smoke_tests/`, `rpk-installer/test/smoke_tests/`, `unified-backend/test/smoke_test/`, `app-installers/test/smoke_tests/`.

Use Pattern B when:
- The package's job is to install/uninstall things, and "smoke" really means "the install pipeline still works on a real package".
- You need C++ scaffolding (test users, backup/restore of system paths, signal handlers) — the kind of thing a shell script can't reasonably express.
- Smoke is a *suite* of `TEST_F`s, not a single check, but still narrower than full unit-test coverage.

### Pattern C — single `smoke_test.cc` inside `unit_tests/` (rare)

A single `.cc` file added to the existing unittests target that validates one critical-path invariant of the package (e.g. "the DB schema matches what the runtime expects"). Reference: `capmgr/src/unit_tests/smoke_test.cc`.

Use Pattern C only when:
- The package has no installer pipeline and no separate test binary, but does have one critical invariant whose failure would brick the package.
- Adding a separate target is overkill — bolting a single `TEST_F` onto the unit-tests binary is enough.

If you're not sure, default to **Pattern A**.

## Phase 1 — Plan the smoke (mandatory, before any code)

Smoke tests have an inverse problem from unit tests: it's easy to write *too much*. The discipline is to pick the narrowest possible signal that catches the broadest set of install/regression failures. Output the worksheet to the conversation before writing any code.

### 1.1 The single question to answer

```
What single failure mode does this smoke test protect against?

  Examples:
   - "the *_unittests binary segfaults on launch because a .so the spec file
      forgot to package isn't on the device" — Pattern A is plenty
   - "the wgt installer no longer accepts a normal .wgt because a manifest-parser
      contract regressed" — Pattern B with one TEST_F per package type
   - "the capmgr DB schema diverged from the runtime expectations" — Pattern C
```

If you can't write the failure mode in one sentence, go narrower. A smoke test that "checks everything works" reliably devolves into a slow, flaky duplicate of the unit-test suite.

### 1.2 Pick the signal

```
Signal:           <what the script/binary asserts on>
Pass criterion:   <exit code 0 / RUN_ALL_TESTS returns 0 / specific log line>
Fail criterion:   <non-zero exit / specific gtest failure / specific assertion>
Time budget:      <upper bound — should be seconds, not minutes>
Boundary:         <in-process / on-disk file / sub-process / real install>
```

For Pattern A the typical answer is: invoke `*_unittests` (or a subset filter), pass if exit 0, time budget 30s. For Pattern B: install + uninstall a sample package, pass if both return `Result::OK`, time budget 60s.

### 1.3 Sanity checklist before moving on

- The smoke test's *name* communicates what regression it catches (e.g. `run-unittest.sh` for the post-install runner; `InstallationMode` for "wgt install works"). Don't name it `Test1`.
- The total runtime is bounded — either by a `timeout` shell command in Pattern A, or by `--gtest_break_on_failure` in Pattern B.
- Cleanup on the device is real: any sample app installed gets uninstalled in `TearDown`; any temp file gets removed; any backup gets restored.
- The check does **not** depend on network access, on a specific user being signed in, or on a system service being in a non-default state — those make CI flakes that look like real bugs.
- Pattern A scripts are POSIX `/bin/sh`-compatible (no `bash`isms), because the device shell is `busybox sh` on minimal Tizen images.

## Phase 2 — Pattern A: write the post-install runner

This is the dominant pattern. The runner is generated **inside the `.spec` file** via a heredoc, then installed under `%{_bindir}/tizen-unittests/%{name}/run-unittest.sh`. Tizen CI auto-discovers it.

### 2.1 Where it goes in the `.spec`

Add the heredoc inside the `%install` section, after `%make_install` and after any gcov-related copy block. The canonical shape (from `notification/packaging/notification.spec`):

```spec
%install
rm -rf %{buildroot}
%make_install

cat << EOF > run-unittest.sh
#!/bin/sh
setup() {
    echo "setup start"
}

test_main() {
    echo "test_main start"
    /usr/bin/<NAME>-unittests
}

teardown() {
    echo "teardown start"
}

main() {
    setup
    test_main
    teardown
}

main "\$*"
EOF

mkdir -p %{buildroot}%{_bindir}/tizen-unittests/%{name}
install -m 0755 run-unittest.sh %{buildroot}%{_bindir}/tizen-unittests/%{name}/
sed -i -e 's/<NAME>/notification/g' %{buildroot}%{_bindir}/tizen-unittests/%{name}/run-unittest.sh
```

Two important details:

- **The `<NAME>` placeholder + `sed -i`.** The pattern is templated so multi-sub-package projects (like `bundle`, which ships `bundle`, `parcel`, `tizen-database`, `tizen-dlog`, `tizen-libopener`, `tizen-shared-queue` from one source tree) can install the same script into multiple `tizen-unittests/<sub>/` paths and `sed` the binary name in. For a single-package project, you can either keep the `<NAME>`/`sed` indirection or hardcode the binary path — both are fine.
- **Escaping `$*`.** The heredoc is unquoted, so `$*` would be expanded by RPM. `\$*` keeps the literal `"$@"`-style passthrough in the generated script.

### 2.2 Add to the `%files` list

The runner usually ships in the same sub-package as the unittests binary it wraps. Add this line to `%files unittests` (or whatever the existing test sub-package is called):

```spec
%files unittests
%{_bindir}/<pkg>_unittests
%{_bindir}/tizen-unittests/%{name}/run-unittest.sh    # <-- new
```

Some packages (e.g. `aul-1`) use `%attr(0755,root,root)` to be explicit about permissions:

```spec
%attr(0755,root,root) %{_bindir}/tizen-unittests/%{name}/run-unittest.sh
```

That's optional but matches a common style.

### 2.3 Optional: smoke-only filter

If `*_unittests` is too slow or has tests that aren't safe to run as smoke (e.g. tests that touch real DBs), gate the runner by a gtest filter:

```sh
test_main() {
    /usr/bin/<NAME>-unittests --gtest_filter='*Smoke*:*BasicSanity*'
}
```

Then prefix the smoke-relevant tests in the C++ source with `Smoke`/`BasicSanity` so the filter picks them up. Keep the filter narrow — once it grows past three patterns, you're rebuilding the unit-test suite and the filter has stopped pulling its weight.

### 2.4 What goes in `setup`/`teardown`

The skeleton has empty `setup`/`teardown` for a reason — most packages don't need them. Add to `setup` only when the test binary itself can't bring up the precondition:

- Permissions/SMACK labels — see `rpc-port`/`bundle`'s `set_perm` helper that calls `chsmack`. Required when the binary writes to `/tmp/home/` or similar privileged paths.
- Test fixtures on disk — e.g. `mkdir /tmp/rpc-port-certs && cp %{_datadir}/rpc-port-unittests/certs/* /tmp/rpc-port-certs/`.
- Environment variables — e.g. `export GCOV_PREFIX=/tmp` for coverage builds.

Mirror every `setup` action in `teardown`. The runner is invoked by CI in a loop, so leftover state poisons the next run.

## Phase 3 — Pattern B: write the C++ smoke gtest

For installer-style packages (wgt-backend, tpk-backend, rpk-installer, unified-backend, app-installers), copy the shape from a sibling package — `wgt-backend/test/smoke_tests/` is the clearest reference.

### 3.1 Directory layout

```
<pkg>/test/smoke_tests/
  ├── CMakeLists.txt
  ├── smoke_test.cc                  # the gtest binary (TEST_F definitions)
  ├── smoke_test_helper.cc           # auxiliary process helper, if needed
  ├── <pkg>_smoke_utils.h
  ├── <pkg>_smoke_utils.cc           # backend interfaces, BackupPath/RestorePath, etc.
  ├── extensive_smoke_test.cc        # the larger sibling — optional
  └── test_samples/                  # real .wgt/.tpk/.rpk fixture packages
```

`test_samples/` ships separately under `%{_datadir}/<pkg>-installer-ut/test_samples/`, and the gtest source addresses them via `kSmokePackagesDirectory`.

### 3.2 The Environment + Test fixture pattern

Smoke for installers needs system-level setup (test user, path backups). The `testing::Environment` subclass owns it:

```cpp
class SmokeEnvironment : public testing::Environment {
 public:
  void SetUp() override {
    if (request_mode_ == ci::RequestMode::USER)
      ASSERT_TRUE(AddTestUser(&test_user));
    backups_ = SetupBackupDirectories(test_user.uid);
    for (auto& path : backups_) ASSERT_TRUE(BackupPath(path));
    CreateDatabase();
  }
  void TearDown() override {
    UninstallAllSmokeApps(request_mode_, test_user.uid, &backend);
    for (auto& path : backups_) ASSERT_TRUE(RestorePath(path));
    if (request_mode_ == ci::RequestMode::USER) ASSERT_TRUE(DeleteTestUser());
  }
  // ...
};
```

Two non-obvious rules:

- **Install a signal handler that calls `env->TearDown()`**. If the test crashes mid-install, the device is left in an inconsistent state (test user still exists, system paths still backed up to `*.bck`). `signalHandler(int signum) { env->TearDown(); exit(signum); }` catches SIGINT/SIGTERM/SIGABRT and unwinds. See `wgt-backend/test/smoke_tests/smoke_test.cc:55–61`.
- **Smoke tests run with elevated capabilities.** The spec file applies `setcap cap_chown,cap_dac_override,cap_fowner,cap_mac_override=eip` to the smoke binary in `%post`. Without this the install/uninstall calls fail with EPERM and look like a code regression. Match the existing convention.

### 3.3 The TEST_F itself

Keep each `TEST_F` to one install/uninstall round trip with one assertion shape:

```cpp
TEST_F(SmokeTest, InstallationMode) {
  fs::path path = kSmokePackagesDirectory / "InstallationMode.wgt";
  std::string pkgid = "smokewgt03";
  ASSERT_EQ(backend.Install(path), ci::AppInstaller::Result::OK);
  ASSERT_TRUE(ValidatePackage(pkgid, env->test_user.uid));
  ASSERT_EQ(backend.Uninstall(pkgid), ci::AppInstaller::Result::OK);
}
```

Don't pile multiple package types into one `TEST_F`. The fault isolation of one-test-one-package is the whole point.

### 3.4 CMake wiring

`<pkg>/test/smoke_tests/CMakeLists.txt` follows the shape in `wgt-backend/test/smoke_tests/CMakeLists.txt`:

- `ADD_EXECUTABLE(${TARGET_SMOKE_TEST} smoke_test.cc)` — one target per smoke binary.
- `ADD_LIBRARY(${TARGET_<PKG>_SMOKE_UTILS} SHARED <pkg>_smoke_utils.cc)` — the package-specific helpers.
- `INSTALL(TARGETS ${TARGET_SMOKE_TEST} DESTINATION ${BINDIR}/<pkg>-installer-ut)` — out-of-line dir matching the `%files` block.
- `INSTALL(DIRECTORY test_samples/ DESTINATION ${SHAREDIR}/<pkg>-installer-ut/test_samples)` — sample artefacts.

Spec-side, list every binary and apply `setcap` in `%post`:

```spec
%post -n %{name}-installer-ut
/usr/sbin/setcap cap_chown,cap_dac_override,cap_fowner,cap_mac_override=eip %{_bindir}/<pkg>-installer-ut/smoke-test
```

### 3.5 What NOT to do in Pattern B

- **No mocks of the installer pipeline.** The whole point is to exercise real `Install`/`Uninstall`. Mocks here turn smoke into a unit test and lose the failure modes it's protecting against.
- **No reliance on a specific device's package state.** Every `TEST_F` installs the package it tests and uninstalls before exiting (or registers cleanup with the `Environment`).
- **No smoke binaries that don't get `setcap`-ed in `%post`.** Without elevated caps the install fails with permission errors and the failure looks like a logic bug.
- **No samples checked into git as binary `.wgt`s without a regeneration script.** Either ship the source manifest + builder script under `test_samples/` so QA can rebuild, or document the SHA in a README.

## Phase 4 — Pattern C: bolt one TEST_F onto unit_tests/

The simplest pattern. Add a single `smoke_test.cc` to the existing `unit_tests/` directory, with one or two `TEST` / `TEST_F` cases that pin the most fragile invariant of the package. CMake picks it up via the existing `AUX_SOURCE_DIRECTORY`.

When this is the right call:
- The package has no installer pipeline (Pattern B doesn't apply).
- A `run-unittest.sh` already exists or is being added (Pattern A is happening separately).
- There's one specific invariant — schema match, capability lookup, init/shutdown round trip — whose regression would be catastrophic and which you want pinned even if the rest of the unit-test suite degrades.

The reference is `capmgr/src/unit_tests/smoke_test.cc`: it backs up the live DB, recreates it from the schema the runtime currently expects, runs a single capability lookup, then restores the backup. One file, two `TEST_F`s, no extra build wiring.

## Phase 5 — Run it

Don't run automatically — tell the user the commands.

For Pattern A, after `gbs build` and install on the device:

```bash
sdb root on
sdb shell /usr/bin/tizen-unittests/<pkg>/run-unittest.sh
echo $?    # 0 = pass
```

For Pattern B:

```bash
sdb root on
sdb shell /usr/bin/<pkg>-installer-ut/smoke-test --gtest_break_on_failure
```

Both should complete inside the time budget from the worksheet. If the runner exits non-zero, look at `dlogutil -d` for the package's tag — the smoke runner is intentionally minimal, so the actual diagnostic almost always lives in dlog, not in the runner's stdout.

## Workflow summary (what to do when invoked)

When the user says "add smoke test for `<pkg>`" or "scaffold smoke for `<pkg>`":

1. **Inspect** the package's `packaging/<pkg>.spec` — does it already have a `cat << EOF > run-unittest.sh` block? If yes, you're in Pattern A; the user probably wants to extend or fix it. If no, propose Pattern A by default.
2. **Inspect** `<pkg>/test/` — is there a `smoke_tests/` directory or installer-style code (look for `Install(`, `Uninstall(`, `BackendInterface`)? If yes, propose Pattern B.
3. **Output** the Phase 1 worksheet — one-sentence failure mode, signal, pass/fail criterion, time budget. Confirm against the section 1.3 checklist.
4. **Generate** the artefacts:
   - Pattern A: heredoc snippet for `%install` + `%files unittests` line.
   - Pattern B: `test/smoke_tests/{smoke_test.cc, CMakeLists.txt, <pkg>_smoke_utils.{h,cc}}`, parent `test/CMakeLists.txt` edit, top-level `CMakeLists.txt` target var, `.spec` `%package`/`%files`/`%post` edits.
   - Pattern C: a single `smoke_test.cc` dropped into the existing `unit_tests/` dir.
5. **Tell the user** the build command and the on-device run command. Do not run them unless asked.

If the user pushes back ("just write the smoke test, skip the worksheet"), explain briefly: smoke tests without a written-down failure mode reliably grow into slow yellow-on-CI duplicates of the unit suite. Offer a one-line worksheet instead of skipping.
